1. Concept Thesis

The VeilPiercer Parallax Engine is a defensive tracking system that fuses three information layers:

1. Radar / minimap contact data
2. Main-view visual object tracking
3. Own-ship telemetry and movement history

The goal is to convert a visible object or radar marker into a confidence-scored positional coordinate estimate.

In a spacecraft interface, the radar/minimap may show simplified relative positions, while the main screen shows objects visually without exposing exact coordinates. A secondary tracking system can compare both views against the ship’s own motion and orientation to infer where objects are actually located.

This builds naturally on the VeilPiercer Relay Mesh idea: one sensor view may be uncertain, but multiple synchronized reference views can form a stronger track through triangulation, time continuity, and confidence scoring.

2. Core Hypothesis

If a spacecraft can observe:

- where a contact appears on radar/minimap,
- where that same contact appears in the visual field,
- how the contact moves over time,
- and how the ship itself moves or rotates,

then the system can estimate the contact’s real position, velocity, and uncertainty volume.

The minimap gives relative tactical geometry.

The visual tracker gives bearing, apparent motion, and object shape.

The ship telemetry gives the reference frame.

Together, they produce a resolved contact track.

minimap position
+ visual bearing
+ own-ship motion
+ time-series movement
= estimated 3D contact coordinates
3. Why Parallax Matters

A single frame can show direction, but it often cannot prove distance.

Multiple frames create parallax. As the ship moves, nearby objects shift faster in the visual field than distant objects. By comparing that apparent motion against known ship movement, the system can refine range.

Object appears left of center.
Ship moves right.
Object shifts across screen.
Radar dot also shifts on minimap.
Fusion engine compares both motions.
Range and position estimate improve.

This matches the 5D tracking idea in the VeilPiercer Relay Mesh: location is not enough by itself. A stronger state estimate includes position, time continuity, and orientation-relative observation quality.

6. Contact Correlation

The system must match radar dots to visible objects.

Example:

Radar contact A:
bearing: 42 degrees right
range: medium
movement: drifting upward-left on minimap

Visual object 3:
screen position: upper-right quadrant
bearing estimate: 40–45 degrees right
motion: drifting upward-left relative to camera

Correlation result:
Radar contact A likely equals visual object 3
confidence: medium-high

The correlator uses:

- bearing similarity
- movement similarity
- timing alignment
- expected camera field of view
- object persistence
- minimap/radar update rate

If two radar contacts are close together, confidence drops until motion separates them.

7. Parallax Fusion Logic

The fusion engine compares object motion against ship motion.

Known:
ship moved from position S1 to S2
camera rotated from angle C1 to C2
object appeared at screen point P1, then P2
radar marker moved from R1 to R2

Solve:
best-fit object location O
best-fit object velocity V
uncertainty volume U

The output is not a magic perfect coordinate. It is a confidence-scored estimate.

Contact ID: VPX-013
Estimated position: x, y, z
Estimated velocity: vx, vy, vz
Uncertainty: 120 meters
Confidence: High
Last updated: T+04.2 seconds

10. Conceptual Control Logic
def veilpiercer_parallax_step(ship, radar, vision, track_store, command):
    """
    High-level conceptual control loop for the VeilPiercer Parallax Engine.
    Defensive coordinate fusion only.
    """

    # 1. Read own-ship state
    ship_state = {
        "position": ship.position,
        "velocity": ship.velocity,
        "orientation": ship.orientation,
        "camera_fov": ship.camera.fov,
        "camera_orientation": ship.camera.orientation,
        "timestamp": ship.time,
    }

    # 2. Read radar/minimap contacts
    radar_contacts = radar.get_contacts()

    # 3. Read visual objects
    visual_objects = vision.detect_and_track_objects()

    # 4. Correlate radar contacts with visual objects
    candidate_pairs = correlate_contacts(
        radar_contacts=radar_contacts,
        visual_objects=visual_objects,
        ship_state=ship_state,
    )

    resolved_tracks = []

    for pair in candidate_pairs:
        radar_contact = pair.radar_contact
        visual_object = pair.visual_object

        # 5. Retrieve previous track history
        history = track_store.get_history(pair.track_id)

        # 6. Estimate coordinate using parallax and time continuity
        estimate = solve_parallax_position(
            ship_state=ship_state,
            radar_contact=radar_contact,
            visual_object=visual_object,
            history=history,
        )

        # 7. Score confidence
        confidence = score_track_confidence(
            radar_agreement=pair.radar_visual_alignment,
            visual_persistence=visual_object.persistence,
            parallax_stability=estimate.stability,
            history_length=len(history),
            uncertainty=estimate.uncertainty,
        )

        # 8. Update track
        track = {
            "id": pair.track_id,
            "position": estimate.position,
            "velocity": estimate.velocity,
            "uncertainty": estimate.uncertainty,
            "confidence": confidence,
            "last_seen": ship_state["timestamp"],
            "source": "radar_visual_parallax_fusion",
        }

        track_store.update(track)
        resolved_tracks.append(track)

    # 9. Return operator-safe output
    return render_operator_layer(
        tracks=resolved_tracks,
        viewer_role=command.viewer_role,
        policy=command.visibility_policy,
    )
11. Mathematical Sketch
Bearing from visual field
bearing ≈ screen_offset × camera_field_of_view
Radar-relative position
x = range × cos(bearing)
y = range × sin(bearing)
Parallax refinement

If the ship moves by a known baseline:

baseline = S2 - S1

and the object’s apparent angle changes:

angle_delta = θ2 - θ1

then range can be refined by triangulation.

Simplified:

range ≈ baseline / tan(angle_delta)

In practice, this would use filtering rather than a single equation.

Useful filters:

- Kalman filter
- particle filter
- optical flow model
- multi-hypothesis tracker
- confidence-weighted sensor fusion
12. Failure Modes
12.1 False Correlation

The system matches the wrong radar dot to the wrong visual object.

Causes:

- cluttered radar field
- overlapping contacts
- similar motion
- poor camera angle

Mitigation:

- require multi-frame agreement
- lower confidence
- preserve alternate hypotheses
12.2 Minimap Distortion

The radar/minimap is symbolic, scaled, or intentionally simplified.

Mitigation:

- calibrate map scale
- detect range rings
- compare motion rather than trusting icon position absolutely
12.3 Camera Motion Confusion

The visual tracker mistakes ship rotation for object movement.

Mitigation:

- subtract own-ship rotation
- stabilize visual frame
- use inertial telemetry
12.4 Low-Signature Contact

The object appears intermittently or weakly.

Mitigation:

- use persistence ribbons
- maintain uncertainty volume
- do not overclaim exact coordinates
12.5 Compartmented Data Conflict

A contact exists in the safety backbone but is redacted for the current viewer.

Mitigation:

- preserve collision safety
- display protected contact marker
- restrict identity and mission metadata

This follows the compartmented visibility principle: do not erase the asset from safety awareness; control how much detail different viewers can resolve.
